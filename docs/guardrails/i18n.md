# i18n Guardrail

> **Status: Active contract** — 覆盖 typed dictionary、fallback、组件内成对文案和高风险产品承诺。
> **为什么先读**：UI 文案既是界面内容也是产品合同。漏翻译会让一种语言下功能失语；把兼容接入、额度或安全能力翻成更强承诺则是产品 bug。

## 词汇表

- `TranslationKey` — 由英文字典 key 推导的联合类型；中文字典必须实现同一 key 集合。
- `t(key, params)` — React 侧翻译函数。
- `translate(locale, key, params)` — 非 React 模块翻译入口。
- fallback chain — 目标语言 → 英文 → raw key；只用于容错，不能作为漏 key 的正常路径。
- local bilingual pair — 局部组件因动态/一次性文案使用的 `isZh ? zh : en` 或 `{description, descriptionZh}` 成对字段。

## 不变量 / 契约表

| # | 不变量 | 由谁守 |
|---|---|---|
| 1 | `en.ts` 是 key schema；`zh.ts` 必须是 `Record<TranslationKey, string>`，新增/删除 key 两边同步 | `src/i18n/en.ts`, `src/i18n/zh.ts` |
| 2 | 参数插值两种语言使用同一占位符集合；不能一边写 `{{name}}` 另一边丢失 | dictionaries + `translate()` |
| 3 | 公共/重复文案进入 typed dictionary；局部动态文案允许 bilingual pair，但必须在同一代码分支内并列，不能只写一种语言 | UI components |
| 4 | 用户可见的 provider/model/套餐名称与 wire ID 分开：品牌名可本地化，model ID、preset key、endpoint 不翻译 | catalog + Settings UI |
| 5 | 风险、安全、计费、额度、兼容性文案中英文必须承诺同一事实；没有 source 的 quota/百分比/订阅名两边都不得显示 | Provider/OAuth UI |
| 6 | 不用某个翻译结果（例如 `t('nav.chats')`）推断 locale 新功能；新代码应从 i18n context 取得明确 locale。存量 workaround 改动时逐步迁移 | I18nProvider consumers |

## 改动检查表

- [ ] 新增/删除 typed key 时同步 `en.ts` 和 `zh.ts`，并检查参数占位符集合
- [ ] 文案若会在多个组件复用，放入 dictionary，不复制多个 `isZh` 分支
- [ ] 局部 bilingual pair 的中英文都由同一状态/事实源驱动
- [ ] model ID、preset key、URL、错误 code 保持原文；只翻译说明与标签
- [ ] 安全/计费/额度/兼容声明没有被任何语言夸大；未知值诚实省略
- [ ] 新增 catalog copy 时同时提供 `description/descriptionZh`、必要的 terms/usage notes 双语字段
- [ ] 跑 `npx tsc --noEmit` 与对应 source-contract/UI test

## 常见坑

- 只在 `en.ts` 加 key：中文类型检查或运行时 fallback 把英文漏到中文 UI。
- 两种语言插值参数不一致：一种语言展示未替换的 `{{value}}`。
- 翻译 wire ID：把 `grok-4.5` / `qwen3.8-max-preview` 改成展示名后又回传 API，导致请求失败。
- 英文说“compatible/可能受上游策略影响”，中文却写成“官方登录/稳定支持”；这是承诺升级，不是翻译润色。
- 没有真实 source 却为了版面完整填 `0%`、`Unlimited` 或猜测套餐名。
- 用某个文案是否等于中文来猜 locale；翻译调整后判断会静默失效。
